1. Qué es y para qué sirve

fs-test-env monta un entorno de pruebas aislado de FacturaScripts (su propio core y su propia base de datos, mesafs_test) y ofrece una web para lanzar los tests de tus plugins y ver los resultados sin tocar la instalación ni la base de datos de trabajo del proyecto.

Con el runner web puedes:

  • Ver, por plugin, todos los ficheros *Test.php con una descripción legible de qué comprueba cada uno.
  • Ejecutar los tests a distintos niveles (todo el plugin, una carpeta, un fichero) y ver el resultado al instante.
  • Consultar el código fuente de cada test sin salir de la página.

2. Cómo acceder

Una vez levantado y provisionado el entorno (ver README.md), la web se sirve en el host configurado en TESTENV_HOST (p. ej. http://test.mesafs.asermar.com). Ábrelo en el navegador: no necesita login.

¿No carga? Levanta el contenedor del entorno con test-bin/bin/up.sh (idempotente: si ya está corriendo, no hace nada). Los detalles de instalación y configuración están en el README.md del repositorio.

3. La pantalla de listado

Al abrir la web verás la cabecera con el título y un resumen (nº de plugins, nº de ficheros de test, versión del core y del entorno), la barra lateral con los plugins, y el panel central con los tests del plugin seleccionado.

La barra lateral

  • FacturaScripts Core aparece siempre primero; su badge es el nº total de tests del core.
  • Cada plugin muestra su nombre y versión y un badge con el número de ficheros de test.
  • Haz clic en un plugin para cargar sus tests en el panel central.

Escenarios (carpetas de Test/)

Los tests de un plugin se agrupan por escenario — cada subcarpeta de Test/ con su propio juego de plugins activos. La cabecera del grupo indica las dependencias que ese escenario activa, p. ej. main · deps: OpenServBus o with-csvimport · deps: OpenServBus,CSVimport. Así conviven tests que requieren un plugin con tests que comprueban su ausencia.

4. Ejecutar tests: los cuatro modos

Puedes lanzar los tests a cuatro niveles, de mayor a menor alcance:

  • ▶ Ejecutar todos (cabecera del plugin) — todos los tests del Core/Plugin. Puede tardar mucho tiempo.
  • ▶ Ejecutar Tests en Carpeta (N) todos los tests de ese escenario en una sola ejecución. Rápido. Puede tardar mucho.
  • ▶ Fichero a fichero ejecuta la carpeta archivo por archivo: más lento en total, pero el resultado se va actualizando con más frecuencia.
  • ▶ Ejecutar (en la tarjeta) — solo ese fichero de test. Lo habitual mientras desarrollas.
Mientras hay una ejecución en marcha, la cabecera muestra ⟳ Ejecutando… y una barra de progreso azul en la parte superior. Con ⊕ Expandir Tests /⊖ Colapsar Tests abres o cierras todas las tarjetas de un escenario.

5. Leer los resultados

Al ejecutar, la tarjeta despliega un panel «Resumen y salida de PHPUnit» con las cifras de la ejecución y, debajo, el estado de cada método de test. La cabecera de la tarjeta muestra además un resumen tipo 3/3 OK · 9 ms.

Qué significa cada cifra

  • OK — tests que pasaron. Fallos — aserciones incumplidas o errores. Omitidos — tests marcados skipped.
  • Aserciones — nº total de comprobaciones. ms — tiempo de la ejecución.
  • ▸ Salida cruda de PHPUnit — despliega la salida original de PHPUnit (útil para depurar un fallo).
  • Cada método muestra su badge OK/FAIL, nombre, nº de aserciones y tiempo.
🧹 Limpiar oculta y vacía todos los paneles de resultados y reinicia los contadores, dejando el listado como al principio.

6. Ver el código de un test

El botón </> Ver código de cada tarjeta despliega el código fuente completo del *Test.php (resaltado), en el mismo panel de la tarjeta, sin abandonar la página. Útil para revisar qué hace exactamente un test antes de ejecutarlo.

7. De dónde salen las descripciones

Los textos legibles de cada tarjeta y de cada método no son mágicos: salen del docblock del propio *Test.php. El runner los renderiza como markdown:

  • Si el docblock tiene un bloque @description, se usa solo ese texto (hasta el siguiente @tag).
  • Si no, se usa todo el docblock (ignorando líneas de @tag).
  • Si la descripción de la clase empieza por un encabezado markdown (## …), ese texto se convierte en el título de la tarjeta (con el nombre del fichero entre paréntesis).
  • Cada método de test puede llevar su propia descripción en su docblock.

El detalle de esta convención, con ejemplos de docblock, está en el README.md.

8. Alternativa por línea de comandos

Si prefieres la terminal, los mismos tests se ejecutan con PHPUnit:

cd <TESTENV_DIR>            # p. ej. test-env/facturascripts
vendor/bin/phpunit Plugins/<Plugin>/Test

El runner web es solo una capa cómoda sobre esta misma ejecución; ambos corren contra el mismo entorno de pruebas provisionado.
